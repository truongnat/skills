# Failure Modes in Brainstorming

## Overview

Brainstorming sessions can fail in predictable ways. Recognizing these patterns allows mitigation and recovery.

## Primary Failure Modes

### 1. Groupthink

**Pattern:** Team converges on first reasonable idea without exploring alternatives.

**Symptoms:**
- First suggestion becomes the default
- Dissent is discouraged or not voiced
- Questions about alternatives are dismissed
- "We all agree" stated without exploration

**Causes:**
- Hierarchical pressure (senior person speaks first)
- Time pressure ("we need to decide now")
- Social cohesion preference over critical thinking
- Fatigue from previous failed explorations

**Mitigation:**
- Use Socratic questioning to force alternatives
- Explicitly require 3+ distinct approaches
- Have junior team members speak first
- Separate idea generation from evaluation phases

**Recovery:**
- "What's a completely different way to solve this?"
- "What if the first idea were not an option?"
- Re-run divergence phase with constraint removal

### 2. Premature Convergence

**Pattern:** Filtering ideas before sufficient exploration.

**Symptoms:**
- Ideas dismissed as "not realistic" without exploration
- Convergence criteria applied before divergence complete
- Feasibility judgments made without data
- "That won't work" with no testing

**Causes:**
- Expert bias ("I know this won't work")
- Risk aversion
- Efficiency mindset over exploration
- Lack of time for proper divergence

**Mitigation:**
- Explicit divergence phase with no filtering
- Document all ideas before any evaluation
- Use "yes, and" instead of "no, but"
- Set minimum idea count before convergence

**Recovery:**
- Return to divergence with new question framing
- Explore why ideas were dismissed
- Test feasibility of dismissed options

### 3. Solution-First Thinking

**Pattern:** Starting with a solution and working backward to justify it.

**Symptoms:**
- Conversation starts with "we should use [technology/approach]"
- Problem statement emerges after solution is proposed
- Questions focus on "how to implement" not "whether to implement"
- Assumptions shaped to fit chosen solution

**Causes:**
- Recent success with similar solution
- Technology bias
- Solution-oriented culture
- Unclear problem statement

**Mitigation:**
- Start with problem/opportunity statement
- Use Socratic questioning to validate problem framing
- Require explicit problem definition before solutions
- Document solution-agnostic success criteria

**Recovery:**
- "What problem is this solution actually solving?"
- "What would success look like independent of this solution?"
- Re-frame from solution space to problem space

### 4. Analysis Paralysis

**Pattern:** Over-exploration without convergence.

**Symptoms:**
- Endless generation of alternatives
- Refusal to filter or prioritize
- "We need more information" as indefinite stall
- Exploration loops without decision criteria

**Causes:**
- Fear of wrong choice
- Lack of clear convergence criteria
- Perfectionism
- Unclear decision authority

**Mitigation:**
- Set explicit convergence criteria upfront
- Timebox divergence phase
- Assign decision authority
- Use satisficing vs optimizing trade-offs

**Recovery:**
- Apply convergence criteria to current options
- Make provisional decision with iteration plan
- Document what information would change the decision

### 5. Assumption Blindness

**Pattern:** Critical constraints not surfaced or questioned.

**Symptoms:**
- Constraints treated as immutable without examination
- "That's just how it works" without justification
- Hidden assumptions not documented
- Surprise failures when assumptions prove false

**Causes:**
- Industry norms treated as laws
- Historical precedent without current validity
- Lack of constraint exploration
- Group-level assumptions not questioned

**Mitigation:**
- Explicit assumption inventory process
- Question "what if this constraint were removed?"
- Test assumption validity with small experiments
- Document all assumptions with rationale

**Recovery:**
- "What are we assuming that might not be true?"
- Run assumption-surfacing sequence
- Design experiments to test critical assumptions

### 6. Dominant Personality Effects

**Pattern:** One person's ideas dominate conversation.

**Symptoms:**
- One person does most of talking
- Ideas from others not explored
- Facilitator unable to balance participation
- Group defers to loudest voice

**Causes:**
- Hierarchical dynamics
- Personality differences
- Lack of facilitation skills
- Cultural deference to authority

**Mitigation:**
- Structured turn-taking
- Anonymous idea generation (e.g., sticky notes)
- Junior voices first
- Facilitator actively balances participation

**Recovery:**
- Explicitly ask for quiet voices
- Use written idea capture before discussion
- Rotate question-asking responsibility

## Detection Signals

### Early Warning Signs

- First idea becomes default without alternatives
- "That won't work" without exploration
- Solution proposed before problem clarified
- Exploration loops without convergence
- One voice dominating conversation

### Mid-Session Checkpoints

- Have we generated 3+ distinct approaches?
- Are assumptions documented?
- Are we still in divergence or have we converged?
- Is convergence criteria defined?
- Are all voices being heard?

### Session Health Metrics

- **Idea diversity:** Count of distinct approaches
- **Assumption coverage:** Number of explicit assumptions surfaced
- **Participation balance:** Voice distribution
- **Question quality:** Ratio of assertion to question
- **Convergence readiness:** Criteria defined and applied

## Recovery Strategies

### When Groupthink Detected

1. Explicitly name the pattern: "We're converging quickly on one option"
2. Force alternative generation: "What if that option weren't available?"
3. Rotate question-asking to break dominance patterns
4. Re-run divergence with constraint removal

### When Premature Convergence Detected

1. Pause evaluation phase
2. Return to divergence with new framing
3. Document dismissed ideas for later reconsideration
4. Test feasibility of dismissed options

### When Solution-First Detected

1. Extract underlying problem statement
2. Evaluate if problem is correctly framed
3. Generate alternatives independent of current solution
4. Compare solution to alternatives against criteria

### When Analysis Paralysis Detected

1. Apply convergence criteria to current options
2. Make provisional decision with iteration gates
3. Document what would change the decision
4. Set explicit deadline for decision

## Prevention Checklist

### Before Session

- [ ] Problem/opportunity statement clear
- [ ] Constraints documented
- [ ] Convergence criteria defined
- [ ] Facilitator assigned
- [ ] Timeboxing established

### During Session

- [ ] Divergence phase complete before convergence
- [ ] Assumptions surfaced and documented
- [ ] 3+ distinct approaches generated
- [ ] All voices heard
- [ ] Convergence criteria applied consistently

### After Session

- [ ] Options prioritized with rationale
- [ ] Assumptions validated or flagged for testing
- [ ] Next steps clear
- [ ] Decision authority confirmed
- [ ] Failure modes documented for learning

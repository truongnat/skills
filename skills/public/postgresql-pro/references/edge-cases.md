## MVCC and vacuum

- **Long-running transactions** (open transactions, idle in transaction) hold **xmin** back — blocks vacuum progress → **bloat**.
- **Autovacuum** — tune thresholds on append-heavy vs update-heavy tables; watch `n_dead_tup` in `pg_stat_user_tables`.
- **Wraparound** — rare on modern servers but catastrophic; monitor `datfrozenxid` and vacuum pressure.

## Transactions and isolation

- Default **`READ COMMITTED`** — each statement sees new snapshots; **phantom reads** possible across statements in one transaction.
- **`REPEATABLE READ`** — snapshot for entire transaction; serialization failures possible — **retry** with backoff.
- **`SERIALIZABLE`** — strongest; higher overhead and false positives — use for critical financial invariants when needed.

## Locks and deadlocks

- **`deadlock_timeout`** — detect cycles; application should **retry** idempotent transactions.
- **Lock ordering** — acquire locks in consistent order across services to reduce deadlocks.
- **`ALTER TABLE`** — many variants require **ACCESS EXCLUSIVE**; plan maintenance windows or use `CONCURRENTLY` where available.

## Sequences and identity

- **Gaps** in `SERIAL` / `identity` sequences are normal after rollback or crash — **do not** encode business invariants as “no gaps”.
- **Copy** / bulk insert can exhaust sequences faster if misconfigured — use `GENERATED BY DEFAULT AS IDENTITY` deliberately.

## Replication and consistency

- **Streaming replicas** lag; **read-your-writes** may fail if read traffic hits a replica — route reads or use session stickiness.
- **Synchronous replication** — stronger durability, higher commit latency — tradeoff for RPO.

## Disk and TOAST

- Large values may be **TOAST**-ed — storage and I/O patterns differ; very wide rows can hurt cache efficiency.

## Security and RLS

- **Row Level Security** — see the dedicated guide: [row-level-security.md](row-level-security.md) (policies, `USING` / `WITH CHECK`, bypass, pooling, performance).
- **Least privilege** — separate `migration` role from `app` role; avoid `SUPERUSER` in application.
